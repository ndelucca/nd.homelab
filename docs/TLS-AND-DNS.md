# TLS and DNS — architecture & options

---

## 1. TLS: real Let's Encrypt certs via DNS-01 (implemented)

### How it works

Every app is served with a **real, publicly-trusted** certificate — no browser
warnings — while **nothing is exposed to the internet**. This is possible because
the certs are issued with the **ACME DNS-01 challenge**, which only needs a public
TXT record; the CA never connects to the host, so no inbound port is opened.

- **Domain:** `ndelucca.dedyn.io` — a free public domain hosted at
  [deSEC](https://desec.io). The original `ndelucca-server.com` was *invented*
  (only resolvable inside AdGuard), so no CA could ever sign it. AdGuard resolves
  `*.ndelucca.dedyn.io` **internally** to LAN IPs; the public zone at deSEC is used
  *only* to answer the `_acme-challenge` TXT lookups during issuance/renewal.
- **Client:** [`lego`](https://github.com/go-acme/lego) (single Go binary, native
  `desec` provider). Installed and kept up to date by `roles/acme`.
- **Server (`ndelucca-server`, .10):** a **wildcard** cert
  `*.ndelucca.dedyn.io` (+ apex) terminated by NGINX for all 13 apps. The vhosts
  already key off `nginx_domain`, so no per-app config changes were needed.
- **Printer (`ndelucca-raspberry-printer`, .12):** issues its **own** cert
  `printer.ndelucca.dedyn.io` and serves Mainsail over HTTPS directly
  (`roles/mainsail_tls`). DNS-01 needs no help from the server, so the Pi
  self-renews independently — no cert syncing between hosts.
- **Renewal:** a daily `lego-renew.timer` on each host runs `lego renew` and
  reloads NGINX via `--renew-hook`. Renewals also keep the deSEC domain active.
- **Secret:** the deSEC API token lives in `group_vars/all/vault.yml`
  (`vault_desec_token`), shared by both hosts.

### Layout

| Host | Role(s) | lego data dir | Cert |
|------|---------|---------------|------|
| `ndelucca-server` | `acme` → `nginx` | `/etc/pki/nginx/lego` | `ndelucca.dedyn.io` + `*.ndelucca.dedyn.io` |
| `ndelucca-raspberry-printer` | `acme` → `mainsail_tls` | `/etc/ssl/mainsail` | `printer.ndelucca.dedyn.io` |

### Fallback to self-signed (manual switch)

The server keeps a **self-signed** cert generated by the nginx role as a
bootstrap floor. If issuance ever fails (bad token, no internet, deSEC down), the
`acme` role fails the play **before** NGINX is reconfigured, so nothing breaks.
To recover, flip the switch and re-run:

```yaml
# inventory/group_vars/homeservers/services.yml
nginx_tls_mode: selfsigned        # letsencrypt (default) | selfsigned
```
```bash
ansible-playbook playbooks/site.yml -l ndelucca-server --tags nginx
```

NGINX then serves the self-signed cert with no downtime. Revert to
`letsencrypt` once the issue is fixed.

### Operations

```bash
# Issue / renew on the server (staging first to avoid rate limits):
ansible-playbook playbooks/site.yml -l ndelucca-server --tags acme   # acme_staging: true|false
# Printer:
ansible-playbook playbooks/printers.yml -l ndelucca-raspberry-printer --tags acme,tls
# Force a renewal check now:
sudo systemctl start lego-renew.service && systemctl status lego-renew.timer
# Inspect the cert:
openssl x509 -in /etc/pki/nginx/lego/certificates/ndelucca.dedyn.io.crt -noout -issuer -dates -ext subjectAltName
```

### Notes & hardening

- A public cert for `*.ndelucca.dedyn.io` appears in Certificate Transparency
  logs (the **name** is public; the services are not reachable from outside).
- The shared deSEC token has full DNS control. To reduce blast radius, create a
  **scoped** deSEC token (limited to the `_acme-challenge` records each host
  needs) and replace `vault_desec_token` with per-host tokens.
- `update.dedyn.io` (deSEC dynDNS) is intentionally **not** used — that would
  publish an A record pointing at your public IP. DNS-01 only needs the API token.

## 2. DNS: AdGuard is the LAN's single point of failure

### Current state

AdGuard Home serves **both** DHCP and DNS for the whole LAN. If it goes down,
every device loses name resolution. This is mitigated by the post-boot watchdog
(`service_maintenance` restarts AdGuard), but a hard failure still takes the LAN
offline for resolution.

### Option A — advertise a secondary DNS in DHCP (cheap, recommended)

Hand clients a fallback resolver alongside AdGuard, so a single AdGuard outage
degrades to "ads not filtered" instead of "no internet":

- Add a public resolver (e.g. `1.1.1.1` / `9.9.9.9`) or the router as the
  **secondary** DNS in the DHCP options.
- Caveat: clients may use the secondary opportunistically, so some queries
  bypass filtering even when AdGuard is healthy. Acceptable for resilience; if
  strict filtering matters more than uptime, skip this and rely on the watchdog.

This is a small change in the AdGuard DHCP config (`adguard_dhcp_*` in
`group_vars/homeservers/services.yml`) — add the secondary DNS option.

### Option B — a real second AdGuard instance (more work)

Run a second lightweight AdGuard (e.g. on the Raspberry Pi already in the
inventory) and advertise both as DNS in DHCP. AdGuard Home doesn't natively sync
config between instances, so you'd manage both via this playbook (the config is
already declarative here). Highest resilience; highest effort.

**Recommendation:** start with **Option A** (one DHCP option) for a large
resilience win at near-zero cost; consider Option B only if DNS uptime becomes
critical.
